## ADDED Requirements

### Requirement: Parse My Clippings.txt files
The system SHALL parse `My Clippings.txt` files generated by Kindle e-readers. Each clipping block SHALL be delimited by a line containing `==========`. The parser SHALL extract: book title, author, clipping type (Highlight, Note, Bookmark), page number, location range, date added, and content text.

#### Scenario: Well-formed My Clippings.txt
- **WHEN** a valid `My Clippings.txt` file is parsed
- **THEN** the system returns a list of parsed clippings grouped by book
- **THEN** each clipping contains: bookTitle, author, type, page, location, dateAdded, and content
- **THEN** books are grouped by normalized title (whitespace-trimmed, encoding-normalized)

#### Scenario: Malformed or corrupted file
- **WHEN** a file is not a valid `My Clippings.txt` (no separator lines, empty file, binary data)
- **THEN** the system returns a validation error with a human-readable description
- **THEN** no database writes occur

#### Scenario: File with only bookmarks
- **WHEN** a clippings file contains only Bookmark-type entries and no highlights or notes
- **THEN** the system returns a validation warning indicating no importable content was found
- **THEN** the user MAY choose to skip the import

#### Scenario: Latin-1 encoded file
- **WHEN** a `My Clippings.txt` file is encoded in Latin-1 rather than UTF-8
- **THEN** the system SHALL attempt Latin-1 decoding as a fallback and parse successfully

### Requirement: Preview import before committing
The system SHALL present a preview of the import before writing any data. The preview SHALL show each book with counts of highlights, notes, and how many are new versus already imported.

#### Scenario: First-time import preview
- **WHEN** a user imports a clippings file for the first time
- **THEN** all clippings are shown as "new" in the preview
- **THEN** each book row shows the total highlight and note counts
- **THEN** the import button shows the total new extract count

#### Scenario: Re-import preview with partial overlap
- **WHEN** a user imports a clippings file that contains both previously imported and new clippings
- **THEN** books with existing clippings show "X already imported" and "Y new"
- **THEN** books with only previously imported clippings show "Already up to date" and are visually de-emphasized (collapsed or dimmed)
- **THEN** books with only new clippings show only the new count
- **THEN** the import button shows only the count of genuinely new extracts

#### Scenario: Complete re-import (nothing new)
- **WHEN** a user imports a clippings file where every clipping already exists in the database
- **THEN** the preview shows "All caught up! No new clippings to import."
- **THEN** no import button is shown (only an "OK" or "Close" button)
- **THEN** no database writes occur

### Requirement: Deduplicate on re-import using content hash
The system SHALL NOT create duplicate Extracts when the same clipping is imported multiple times. Deduplication SHALL be based on a content hash derived from the clipping text and its parent book title.

#### Scenario: Re-import of same file
- **WHEN** a user imports the exact same `My Clippings.txt` file that was previously imported
- **THEN** zero new Extracts are created
- **THEN** zero new Documents are created
- **THEN** the user sees the "All caught up" message

#### Scenario: Clippings file with new additions
- **WHEN** a user imports a `My Clippings.txt` file where 5 new clippings have been appended since the last import
- **THEN** only the 5 new Extracts are created
- **THEN** the existing Document for each affected book is updated (extractCount incremented, dateModified refreshed)

#### Scenario: Same book across multiple files
- **WHEN** clippings for the same book appear in two different `My Clippings.txt` files (e.g., from two different Kindles)
- **THEN** clippings are merged into the same Document
- **THEN** only genuinely new clippings (by content hash) are created as Extracts
- **THEN** the Document title and metadata are preserved from the first import

#### Scenario: Near-duplicate highlights (same text, slightly different location)
- **WHEN** a clipping has identical text content to an existing Extract for the same book
- **THEN** it SHALL be treated as a duplicate regardless of location or page number differences
- **THEN** no new Extract is created

### Requirement: Create Documents and Extracts from clippings
The system SHALL create one Document per unique book and one Extract per highlight or note clipping. Bookmark-type clippings SHALL NOT be imported as Extracts.

#### Scenario: Highlight clipping
- **WHEN** a clipping of type "Highlight" is imported
- **THEN** an Extract is created with `content` set to the highlighted text
- **THEN** the Extract's `dateCreated` is set to the parsed Kindle date
- **THEN** the Extract is tagged with `["kindle"]`

#### Scenario: Note clipping
- **WHEN** a clipping of type "Note" is imported
- **THEN** an Extract is created with `content` set to the note text
- **THEN** the Extract's `notes` field contains the note content
- **THEN** the Extract is tagged with `["kindle", "kindle-note"]`

#### Scenario: Bookmark clipping
- **WHEN** a clipping of type "Bookmark" is encountered
- **THEN** no Extract is created
- **THEN** the bookmark is counted separately in the preview (e.g., "23 highlights · 5 notes · 12 bookmarks (skipped)")

#### Scenario: Document creation for a new book
- **WHEN** clippings reference a book not yet in the database
- **THEN** a Document is created with:
  - `title` = the book title (with author in parentheses if parseable)
  - `fileType` = "other"
  - `category` = "Kindle"
  - `tags` = ["kindle-import"]
  - `filePath` = `"kindle://<normalized-title-hash>"` (synthetic, non-filesystem path)
  - `metadata.source` = "kindle-clippings"
  - `metadata.author` = parsed author name (if available)

#### Scenario: Document matching for existing book
- **WHEN** clippings reference a book that already exists as a Document (matched by normalized title)
- **THEN** no new Document is created
- **THEN** new Extracts are added to the existing Document
- **THEN** the existing Document's `extractCount` and `dateModified` are updated

### Requirement: Parse Kindle date format
The system SHALL parse the Kindle date format: `"Added on DayOfWeek, Month DD, YYYY H:MM:SS AM/PM"` (e.g., `"Added on Sunday, January 1, 2024 3:45:22 PM"`).

#### Scenario: Valid Kindle date
- **WHEN** a clipping has a valid date string
- **THEN** the date is parsed into an ISO 8601 timestamp and stored as the Extract's `dateCreated`

#### Scenario: Unparseable date
- **WHEN** a clipping has a date string that cannot be parsed
- **THEN** the system falls back to the file's last modification timestamp
- **THEN** if the file mtime is unavailable, the current time is used
- **THEN** a warning is included in the import result (not shown to the user unless debug mode is on)

### Requirement: Accept files via file picker
The system SHALL provide a "Kindle" tab in the EnhancedFilePicker dialog with a file picker filtered to `.txt` files.

#### Scenario: File picker selection
- **WHEN** a user selects the "Kindle" tab in the import dialog and picks a `.txt` file
- **THEN** the system validates the file as `My Clippings.txt` format
- **THEN** on success, the import preview dialog is shown
- **THEN** on failure, an error message is shown explaining the file doesn't appear to be a Kindle clippings file

### Requirement: Accept files via Import/Export settings
The system SHALL add a "Kindle Clippings" entry in the "Additional Imports" section of Import/Export settings, alongside the existing Zotero/Mendeley and podcast importers.

#### Scenario: Settings page import
- **WHEN** a user clicks the Kindle Clippings import button in settings
- **THEN** a file picker opens filtered to `.txt` files
- **THEN** the same validation and preview flow is used

### Requirement: Post-import success feedback
The system SHALL display a clear success message after import, with actionable navigation.

#### Scenario: Successful import of new clippings
- **WHEN** the import completes with N new Extracts across M books
- **THEN** a toast or inline message shows: "Imported N new highlights and notes from M book(s)"
- **THEN** the message includes a clickable link "View imported books" that navigates to the Documents view filtered to the `kindle-import` tag

#### Scenario: Import with zero new clippings
- **WHEN** the preview determined zero new clippings and the user dismissed the dialog
- **THEN** no toast is shown (the dialog's "All caught up" message is sufficient)

### Requirement: Expose Tauri commands for import
The system SHALL expose three Tauri commands following existing import patterns (path-based, matching Anki/SuperMemo/StudyJSON patterns).

#### Scenario: `validate_kindle_clippings` command
- **WHEN** the frontend calls `validate_kindle_clippings` with a file path
- **THEN** the system parses the file, groups clippings by book, checks existing database for duplicates
- **THEN** returns: total book count, total clipping count, per-book breakdown (title, new highlights, existing highlights, new notes, existing notes), and any warnings
- **THEN** no database writes occur

#### Scenario: `import_kindle_clippings` command
- **WHEN** the frontend calls `import_kindle_clippings` with a file path
- **THEN** the system creates/updates Documents and Extracts for all new clippings
- **THEN** returns: new documents created, new extracts created, books updated, and any warnings
- **THEN** existing Extracts (matched by content hash) are NOT modified

#### Scenario: `parse_kindle_clippings` command
- **WHEN** the frontend calls `parse_kindle_clippings` with a file path
- **THEN** the system parses the file and returns raw parsed data without checking the database
- **THEN** this is used for the initial validation before the full dedup-aware preview
